1. ## **方案定位**

本系统拟设计为一种面向多智能体系统（MAS）的 **agent-native database**。

> 能不能把 agent 的 memory/state/event/artifact 统一成对象系统，并通过事件驱动物化和结构化检索，提升推理相关记忆检索的有效性与效率？

它不是传统关系数据库的简单延伸，也不只是向量数据库加几张 memory 表，而是一个同时支持：

- 对象化记忆管理
- 事件驱动状态演化
- 版本与可见性控制
- 多表示检索
- 结构化证据召回
- 多智能体共享治理
- 推理型原生查询算子
- 云原生弹性执行

的数据库系统。

从工程路线看，系统采用：**Milvus as Data Plane + Manu as Control/Event Plane + Agent-Native Semantic/Governance Layer**

其中，Milvus 提供高性能向量检索、segment、LSM 式动态数据管理、snapshot isolation、异构 CPU/GPU 执行等能力；Manu 提供 access / coordinator / worker / storage 四层架构、WAL/binlog 主干、TSO/time-tick、独立扩缩容 worker 等云原生机制。 

1. ## **总体设计目标**

### **2.1 让数据库存储“智能体认知状态”而非仅存向量**

现有向量数据库通常把对象看作“向量 + 属性”，但 MAS 场景中的关键对象包括：

- 事件轨迹
- 记忆单元
- 当前状态
- 工具调用痕迹
- 共享黑板
- 归纳规则
- 推理链及来源

这些内容不能简化为一张 memory(id, content, embedding, metadata) 表。

### **2.2 让数据库返回“结构化证据”，而不是一组孤立 chunk**

数据库查询不应只返回 top-k 片段，而应返回：

- 证据子图
- 证据子张量
- 版本边界
- provenance
- proof trace

从而让多跳推理、对比推理、约束推理等在 DB 内部由原生算子支撑。

### **2.3 让状态演化由事件流推进，而不是直接覆盖写**

系统中 memory、state、policy 等对象不直接 upsert，而是：

**request → WAL/event log → 异步物化 → canonical object / retrieval view**

这一路线与 Manu 的 log backbone 思想一致：所有改变系统状态的请求进入逻辑日志，再由各类 subscriber 独立消费与更新。

### **2.4 让多智能体共享具有边界、可解释、可审计**

共享记忆不是简单“大家都能读一个 namespace”，而必须内建：

- scope / ACL
- TTL / decay
- quarantine
- conflict / merge
- provenance
- rollback / audit

1. ## **顶层架构**

## **3.1 Data Plane：参考 Milvus**

这一层是系统的**检索与存储内核**，保留 Milvus 式能力：

- vector query
- multi-vector query
- attribute filtering
- segment-based storage
- LSM / MemTable / merge
- snapshot isolation
- CPU/GPU heterogeneous execution
- bufferpool / SSD / object storage 分层缓存

Milvus 的核心特点包括：支持 Query Engine / GPU Engine / Storage Engine 组合，支持多向量对象、属性过滤、图索引和量化索引，采用 LSM 风格管理动态写入，以 segment 为搜索、调度和缓冲的基本单位，并通过 snapshot isolation 保证读写一致视图。

### **Data Plane 在本系统中的职责**

它不直接暴露为“用户可见的 memory 语义接口”，而是负责：

- 向量与混合索引执行
- segment 管理
- 数据 flush / merge / compaction
- snapshot 一致读
- CPU/GPU 协同执行
- 热冷分层缓存
- retrieval view 的底层执行也就是说，它是**执行面**，不是完整的语义面。

## **3.2 Control + Event Plane：参考 Manu**

这一层是系统的**云原生调度与事件中枢**，采用 Manu 风格的分层：

- Access Layer
- Coordinator Layer
- Worker Layer
- Storage Layer
- WAL / binlog / pub-sub backbone
- TSO / time-tick
- 独立扩缩容 worker

Manu 的四层架构非常清晰：access layer 由无状态 proxy 构成，负责接收请求、路由、聚合和元数据校验；coordinator layer 负责管理系统状态和元数据，并协调 query/data/index 等节点；worker layer 由无状态 query/data/index nodes 构成，可独立扩缩容；storage layer 使用 etcd 存状态与元数据，使用 S3/MinIO/文件系统持久化大体量数据和索引。更关键的是，Manu 把 WAL 和 binlog 作为系统主干：WAL 是增量、流式、低延迟的 row-based 日志，binlog 是基础、批式、列式的持久层；logger 写 WAL，data nodes 把 WAL 转成 column-based binlog，query/index 等只读组件作为独立 subscriber 消费日志并维护自己的只读视图。

### **Control/Event Plane 在本系统中的职责**

- 写入事件化
- 对象生命周期协调
- worker 编排
- 元数据管理
- 版本可见性推进
- policy 与共享治理协调
- event-sourcing 与回放

## **3.3 Agent-Native Semantic & Governance Layer**

用于把数据库从“向量检索系统”升级为“智能体认知数据库”。

这一层不是单独加一个物理大层，而是**横跨 coordinator、worker、storage 的语义层**，包含：

- 记忆语义建模
- 状态语义建模
- policy 治理
- share contract
- proof trace
- 冲突与合并
- subtensor / subgraph reasoning operators

1. ## **核心数据建模原则**

## **5.1 从表模型转向“对象 + 事件 + 版本 + 索引”**

系统不应以关系表和 join 为中心，而应以：

- 对象
- 事件
- 版本
- 索引投影

为中心。

Milvus 已经证明，把 entity 视作由一个或多个向量与可选属性组成的对象，并采用列式存储与 segment 管理，是向量系统中有效的建模方式。Manu 则说明，向量数据库不需要关系数据库那种跨表操作，更适合简单对象模型、日志驱动和可调一致性。

因此本系统中：

- 状态不是直接覆盖写，而由事件流推进
- 对象不是单一向量，而是多表示实体
- 查询视图不是唯一真相，而是带版本和时间边界的 materialized view

## **5.2 记忆不是一张表**

错误做法是：memory(id, content, embedding, metadata)

这样的问题是：

- 没有事件来源
- 没有版本
- 没有因果关系
- 没有状态层
- 没有异步物化链
- 没有多索引投影
- 没有共享治理语义

正确做法是把系统拆成对象系统。

1. ## **Tensor 化记忆：作为逻辑抽象**

## **7.1 核心定义**

- tenant_id

- scope

- time_bucket

- version

**tensor cell 的含义：**每个 cell 不是一个数值，而是一个 **structured sparse tensor cell**，内部存：

Tensor 是数据库的逻辑表示与查询抽象；底层物理实现可由 object store、columnar segments、graph edges、vector indexes 等共同支撑。

1. ## **记忆语义三层：Base / Policy / Adaptation**

## **9.1 Base Layer**

保存：

- embedding_base
- 原始文本 / blob
- provenance
- source event
- 写入时间
- 版本戳
- revision lineage

原则：

- append-only
- 支持 time-travel
- correction 通过新版本或 correction edge 表达
- 不直接覆写旧事实

## **9.2 Policy Layer**

这是治理层，负责：

- salience_weight
- ttl
- decay_fn
- confidence
- verified_state
- quarantine_flag
- visibility_policy
- read_acl / write_acl

建议再加：

- policy_reason
- policy_source
- policy_event_id

原则：

- 默认软遗忘，不直接物理删除
- 合规要求下才硬删
- 支持 soft TTL / hard delete window
- 所有 policy 调整可审计

## **9.3 Adaptation Layer**

允许学习的是：

- retrieval profile
- ranking parameters
- filtering thresholds
- projection weights
- embedding family selection

限制：

- 不重写 base embedding
- 不篡改 base payload
- 不修改 provenance
- 必须可审计、可回滚、可禁用

1. ## **原生推理型查询算子**

1. ## **共享治理与冲突处理**

## **12.1 Share Contract**

引入：

### **MemoryShareContract**

字段包括：

- scope
- read_acl
- write_acl
- derive_acl
- ttl_policy
- consistency_level
- merge_policy
- quarantine_policy
- audit_policy

这让共享不是单个 scope 字段，而是完整协议单元。

## **12.2 Conflict & Merge Engine**

冲突是 MAS 中的原生现象，不是异常。

处理三类冲突：

### **Fact conflict**

同一事实多版本并存，不强制合一。

### **Plan conflict**

多个 agent 对计划不一致，可通过权重、投票或仲裁处理。

### **State conflict**

共享状态并发写入，可采用：

- last-writer-wins
- causal merge
- weighted merge
- CRDT-like partial merge

1. ## **版本与时间模型**

统一四种语义：

- event_time：事件实际发生时间
- ingest_time：进入 WAL 时间
- visible_time：对查询可见时间
- logical_version：对象版本号

Manu 通过 TSO 和 hybrid logical clock 生成与物理时间接近的逻辑时间，并用 time semantics 支撑确定性顺序与 bounded staleness。Milvus 则通过 snapshot isolation 让每个查询工作在启动时的快照上，不受后续写入干扰。

本系统据此支持：

- replay
- time-travel query
- bounded staleness
- session-consistent reads
- rollback / audit

1. ## **分片、channel 与 segment 设计**

## **15.1 写入分片键**

优先级：

- tenant_id

- session_id

- WAL channel 按 tenant / agent / session hash

- lineage 尽量与写入流同向

- object_type

- event.tooltrace.2026-03-13.sessionX
- state.plan.current.agentB

Milvus 中 segment 是搜索、调度和缓冲的基本单位；Manu 中 segment 也是数据组织单位，并有 growing / sealed 状态。

1. ## **Micro-batching 执行机制**

这部分放进 Query Worker / Retrieval Executor。

### **17.1 Cross-agent micro-batching**

多个 agent 并发查询时，按：

- query signature
- scope
- time window
- embedding family
- namespace

进行聚类和合批。

### **17.2 Intra-agent chain batching**

单 agent 长思维链中，把相邻检索步骤：

- 合并预取
- 共享候选簇
- 分阶段消费

### **17.3 目的**

- 提升 GPU/CPU 利用率
- 降低重复索引访问
- 减少多 agent 并发时的检索瓶颈
- 为 subtensor/subgraph retrieval 做批式执行优化
